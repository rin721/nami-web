# 质量基线

质量基线说明组件库必须长期守住的行为契约。它不替代组件 API，但会解释为什么 API、metadata、tokens 和浏览器测试需要保持同源。

## 目标

确保每次发布都覆盖单元测试、浏览器测试、manifest、包入口、主题矩阵、可访问性和乱码哨兵，避免官网或组件库在局部改动后悄悄退化。

## 已覆盖契约

| 范围 | 检查点 |
| --- | --- |
| 单元测试 | tokens 派生、theme studio、catalog、metadata、表单和组件状态。 |
| 浏览器测试 | custom elements 注册、主题切换、路由、组件示例、Settings 导出。 |
| 打包契约 | 根入口、单组件入口、register 入口、critical CSS、schema、anatomy JSON。 |
| 文档契约 | Start、Theme、Resources、Components 详情页都有章节导航和本页目录。 |
| i18n | `/zh-CN/` 与 `/en-US/` 路由数量、标题结构和核心章节同步。 |

## 验证流程

```bash
npm test
npm run test:browser
npm run verify
```

<nami-docs-callout variant="warning" title="新增组件时必须同步 metadata">
组件实现、metadata、catalog 示例、custom-elements manifest、anatomy JSON 和官网组件页不能分叉维护。官网组件详情页只消费公共 metadata 与 catalog。
</nami-docs-callout>

## 可访问性

交互组件必须保留键盘路径、focus-visible、ARIA 状态、disabled/loading 阻断、forced-colors 和 reduced-motion。浮层组件还需要检查焦点返回和 escape 行为。

## 常见失败

| 失败 | 排查 |
| --- | --- |
| 浏览器测试里元素没有升级 | 检查组件入口是否被导入，或 register 文件是否输出。 |
| 视觉回归只在 dark 失败 | 检查 semantic token 是否被硬编码颜色绕过。 |
| 文档出现乱码 | 运行乱码哨兵，检查可见文案里是否出现常见编码错位字符或连续问号。 |
