# Semantic Anatomy

Semantic anatomy makes components explainable, testable, and consumable by theme tools. It is not visual commentary; it is part of the public component contract.

## Goal

Every interactive component should expose stable host state, CSS Parts, tokens, slots, events, and anatomy so docs, theme DIY, visual regression, and cross-framework wrappers can share one source.

## Anatomy

Anatomy describes explainable component parts such as `base`, `control`, `label`, `description`, `error`, `icon`, and `content`. The Anatomy section on component pages comes from public metadata instead of MDX tables.

## State Contract

| State | Purpose |
| --- | --- |
| `data-state` | Mutually exclusive states such as open/closed, checked/unchecked, active/inactive. |
| `data-disabled` | Disabled styling and event guards. |
| `data-invalid` | Form validation failure. |
| `data-loading` | Prevent repeated submissions and expose loading state. |
| `data-placement` | Floating placement for tooltip, popover, drawer, and similar components. |

## CSS Parts

CSS Parts are the safe boundary for deep DIY. Component internals can change DOM structure, but if public part names stay stable, user styles and theme tools keep working.

```css
nami-input::part(control) {
  border-color: var(--nami-color-primary);
}
```

## Style Hooks

Style hooks connect part, state, and tokens. For example, `control:invalid` maps to error border tokens. This is more stable than guessing from DOM and easier for theme designers to generate.

## Metadata Source

<nami-docs-steps>
<ol>
<li><strong>Component implementation declares the public contract.</strong>Attributes, events, slots, parts, states, and tokens enter metadata.</li>
<li><strong>Docs consume metadata.</strong>Component detail pages do not duplicate API tables in MDX.</li>
<li><strong>Tooling exports machine-readable files.</strong>Anatomy JSON, theme schema, and custom-elements manifest are generated from the same source.</li>
</ol>
