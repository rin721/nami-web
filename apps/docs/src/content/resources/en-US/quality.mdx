# Quality Baseline

The quality baseline documents the behavior contracts the library must keep over time. It does not replace component APIs, but it explains why API, metadata, tokens, and browser tests must stay in sync.

## Goal

Every release should cover unit tests, browser tests, manifests, package entries, theme matrix, accessibility, and mojibake sentinels so docs and components do not silently regress after local changes.

## Covered Contracts

| Area | Checks |
| --- | --- |
| Unit tests | Token derivation, theme studio, catalog, metadata, forms, and component state. |
| Browser tests | Custom element registration, theme switching, routing, component examples, and Settings export. |
| Package contract | Root entry, single component entries, register entries, critical CSS, schema, and anatomy JSON. |
| Docs contract | Start, Theme, Resources, and component detail pages have section navigation and On this page. |
| i18n | `/zh-CN/` and `/en-US/` route count, title structure, and core sections stay aligned. |

## Verification

```bash
npm test
npm run test:browser
npm run verify
```

<nami-docs-callout variant="warning" title="New components must sync metadata">
Implementation, metadata, catalog examples, custom-elements manifest, anatomy JSON, and docs pages must not drift. Component detail pages consume public metadata and catalog data only.
</nami-docs-callout>

## Accessibility

Interactive components must preserve keyboard paths, focus-visible, ARIA state, disabled/loading guards, forced-colors, and reduced-motion. Overlay components also need focus return and escape behavior checks.

## Common Failures

| Failure | Check |
| --- | --- |
| Element does not upgrade in browser tests | Check component imports or register output. |
| Visual regression only fails in dark mode | Check whether a hard-coded color bypassed semantic tokens. |
| Docs show mojibake | Run sentinels for common encoding artifacts and repeated question marks in visible copy. |
